# AIClient-2-API 技术架构文档

## 1. 项目概述

**AIClient-2-API** 是一个强大的 API 代理中间件，能够将多种 AI 提供商（Gemini、Claude、Grok、Codex、Kiro 等）的 API 统一转换为标准的 OpenAI 兼容接口。

### 1.1 核心特性

- **多进程架构**：主进程监控 Worker 进程，支持崩溃重启
- **策略模式**：各提供商使用独立策略类实现
- **适配器模式**：统一接口封装不同提供商
- **插件架构**：支持认证、监控、统计等扩展
- **提供商池管理**：支持多账号池、智能轮询、故障转移
- **协议转换引擎**：支持 OpenAI、Claude、Gemini 等协议互转

### 1.2 支持的提供商

| 提供商 | 常量 | 协议 | 认证方式 |
|--------|------|------|----------|
| Gemini CLI | `gemini-cli-oauth` | Gemini | OAuth 2.0 |
| Antigravity | `gemini-antigravity` | Gemini | OAuth 2.0 |
| OpenAI Custom | `openai-custom` | OpenAI | API Key |
| OpenAI Responses | `openaiResponses-custom` | OpenAI | API Key |
| Claude Custom | `claude-custom` | Claude | API Key |
| Kiro | `claude-kiro-oauth` | Claude | OAuth 2.0 |
| Qwen | `openai-qwen-oauth` | OpenAI | OAuth 2.0 |
| iFlow | `openai-iflow` | OpenAI | OAuth 2.0 |
| Codex | `openai-codex-oauth` | Codex | OAuth 2.0 |
| Grok | `grok-custom` | OpenAI | Cookie/SSO |
| Local Model | `local-model` | OpenAI | 无 |
| Forward | `forward-api` | OpenAI | 无 |

## 2. 架构设计

### 2.1 多进程架构图

```
┌─────────────────────────────────────────────────────────────────┐
│                        主进程 (Master)                          │
│                    src/core/master.js                           │
│  • 子进程管理（fork/重启）                                      │
│  • 资源监控（CPU/内存阈值告警）                                  │
│  • 管理端口 3100（/master/status 等）                           │
│  • IPC 通信（心跳检测）                                         │
└─────────────────────────┬───────────────────────────────────────┘
                          │ fork() + IPC
                          ▼
┌─────────────────────────────────────────────────────────────────┐
│                      Worker 进程                                 │
│                  src/services/api-server.js                     │
│  • HTTP 服务器（默认 3000 端口）                                 │
│  • 插件系统加载                                                  │
│  • 提供商池初始化                                                │
│  • 请求处理管道                                                  │
└─────────────────────────────────────────────────────────────────┘
```

**关键设计**：
- 主进程 (`master.js`) 负责 fork Worker 子进程，监控其资源使用
- Worker 进程 (`api-server.js`) 处理实际的 HTTP 请求
- 通过 `IS_WORKER_PROCESS` 环境变量区分进程类型
- 使用 IPC（进程间通信）进行心跳检测和状态同步

### 4.2 认证流程

**文件**: `src/core/plugin-manager.js` (第 270-330 行)

```javascript
async executeAuth(req, res, requestUrl, config) {
    const authPlugins = this.getAuthPlugins();  // 获取所有认证插件

    for (const plugin of authPlugins) {
        const result = await plugin.authenticate(req, res, requestUrl, config);

        if (result.handled) return { handled: true, authorized: false };
        if (result.authorized === false) return { handled: true, authorized: false };
        if (result.authorized === true) return { handled: false, authorized: true, data: result.data };
        // authorized === null 表示此插件不处理，继续下一个
    }
    return { handled: false, authorized: false };  // 无插件授权
}
```

### 4.4 API Potluck 插件详解

**文件**: `src/plugins/api-potluck/index.js`

```javascript
// 核心功能
const apiPotluckPlugin = {
    name: 'api-potluck',
    type: 'auth',              // 认证插件
    _priority: 10,             // 优先级高于 default-auth (9999)

    // 认证方法
    async authenticate(req, res, requestUrl, config) {
        const apiKey = extractPotluckKey(req, requestUrl);
        if (!apiKey) return { handled: false, authorized: null };  // 非 potluck 请求

        const validation = await validateKey(apiKey);
        if (!validation.valid) {
            sendPotluckError(res, error);
            return { handled: true, authorized: false };
        }
        return { handled: false, authorized: true, data: { potluckApiKey: apiKey } };
    },

    // 钩子函数 - 记录用量
    hooks: {
        async onContentGenerated(hookContext) {
            const usage = getPendingUsageForHookContext(hookContext);
            await incrementUsage(hookContext.potluckApiKey, hookContext.toProvider, hookContext.model, usage);
        }
    }
};
```

### 5.2 适配器工厂

**文件**: `src/providers/adapter.js` (第 650-690 行)

```javascript
export function getServiceAdapter(config) {
    const providerKey = getServiceInstanceKey(provider, config.uuid);

    // 单例模式：已存在则直接返回
    if (serviceInstances[providerKey]) {
        return serviceInstances[providerKey];
    }

    // 创建新实例
    let AdapterClass = adapterRegistry.get(provider);
    if (!AdapterClass) {
        // 支持前缀匹配（如 openai-custom-1 → openai-custom）
        for (const [key, value] of adapterRegistry.entries()) {
            if (provider.startsWith(key + '-')) {
                AdapterClass = value;
                break;
            }
        }
    }

    serviceInstances[providerKey] = new AdapterClass(config);
    return serviceInstances[providerKey];
}
```

### 6.2 节点选择算法 (LRU 评分)

**文件**: `src/providers/provider-pool-manager.js` (第 603-650 行)

```javascript
_calculateNodeScore(providerStatus, now = Date.now()) {
    // 1. 不健康的节点排最后
    if (!config.isHealthy || config.isDisabled) return 1e18;

    // 2. 并发限制检查
    if (concurrencyLimit > 0 && state.activeCount >= concurrencyLimit) {
        if (queueLimit > 0 && state.waitingCount >= queueLimit) return 1e17;
        return 1e15 + state.waitingCount * 1e10;
    }

    // 3. 新鲜节点判断（60秒内检查过）
    const isFresh = lastHealthCheckTime && (now - lastHealthCheckTime < 60000);

    // 4. 评分计算
    const baseScore = isFresh ? -1e14 : lastUsedTime;  // 新鲜节点负偏移
    const usageScore = usageCount * 10000;              // 使用次数惩罚
    const sequenceScore = relativeSeq * 1000;           // 打破平局
    const loadScore = activeCount * 5000;              // 负载惩罚

    return baseScore + usageScore + sequenceScore + loadScore;
}
```

## 8. 服务管理器

### 8.1 API 服务初始化

**文件**: `src/services/service-manager.js`

```javascript
export async function initApiService(config, isReady = false) {
    // 初始化 ProviderPoolManager
    providerPoolManager = new ProviderPoolManager(config.providerPools || {}, {
        globalConfig: config,
        maxErrorCount: config.MAX_ERROR_COUNT ?? 10,
        providerFallbackChain: config.providerFallbackChain || {},
    });

    // 初始化号池中所有节点（避免首个请求延迟）
    for (const [providerType, providerConfigs] of Object.entries(config.providerPools)) {
        for (const providerConfig of providerConfigs) {
            if (providerConfig.isDisabled) continue;

            const nodeConfig = deepmerge(config, {
                ...providerConfig,
                MODEL_PROVIDER: providerType
            });
            delete nodeConfig.providerPools;

            getServiceAdapter(nodeConfig);  // 触发初始化
        }
    }
}
```

### 8.2 获取 API 服务（含池选择）

**文件**: `src/services/service-manager.js` (第 200-260 行)

```javascript
export async function getApiService(config, requestedModel = null, options = {}) {
    // 1. 前置路由解析（处理 provider:model 前缀）
    const { effectiveProvider, actualModelName } = await _resolveEffectiveRouting(config, requestedModel);

    // 2. 从号池选择节点
    if (providerPoolManager && isPoolable) {
        const selectedProviderConfig = await providerPoolManager.selectProvider(
            config.MODEL_PROVIDER,
            actualModelName,
            { skipUsageCount: true }
        );

        // 合并选中配置到请求 config
        serviceConfig = deepmerge(config, selectedProviderConfig);
    }

    return getServiceAdapter(serviceConfig);
}
```

## 13. 关键设计模式

### 13.1 策略模式
- 各提供商实现使用独立的策略类（`*-strategy.js`）
- 转换器策略（`ClaudeConverter`、`OpenAIConverter` 等）

### 13.2 适配器模式
- `ApiServiceAdapter` 定义统一接口
- 各提供商适配器包装具体实现
- `getServiceAdapter(config)` 工厂方法创建实例

### 13.3 单例模式
- `PluginManager` 单例
- `IPCManager` 单例
- 服务实例缓存（`serviceInstances` 映射）

### 13.4 责任链模式
- 插件认证流程
- 插件中间件流程
- 故障转移链（`providerFallbackChain`）

### 13.5 观察者模式
- SSE 事件广播（`event-broadcast.js`）
- 健康状态变化广播

## 16. 故障转移机制

### 16.1 Provider Fallback Chain

当主提供商类型全部不健康时，自动尝试兼容协议的提供商：

```json
{
  "providerFallbackChain": {
    "gemini-cli-oauth": ["gemini-antigravity"],
    "claude-kiro-oauth": ["claude-custom"]
  }
}
```

### 16.2 Model Fallback Mapping

当请求特定模型不可用时，自动映射到替代模型：

```json
{
  "modelFallbackMapping": {
    "claude-opus-4-5": {
      "targetProviderType": "claude-custom",
      "targetModel": "claude-3-5-sonnet-20241022"
    }
  }
}
```

## 17. 安全特性

- **API Key 认证**：支持多种认证方式（Bearer Token、Header、Query）
- **登录保护**：最大尝试次数、锁定持续时间、最小间隔
- **密码安全**：PBKDF2 + SHA-512，310000 次迭代
- **CORS 控制**：支持跨域请求配置
- **日志脱敏**：敏感信息自动脱敏

## 18. 性能优化

### 18.1 提供商选择并发安全
- 互斥锁确保同一提供商类型不会并发选择同一节点
- 序列号机制确保毫秒级并发也能区分

### 18.2 令牌刷新优化
- 缓冲队列机制（5秒延迟去重）
- 并发控制（全局2个并行，每个提供商内部1个）
- 刷新超时保护（默认60秒）

### 18.3 配置保存优化
- 防抖机制（1秒延迟批量写入）
- 避免频繁文件 I/O

### 18.4 请求处理优化
- 请求上下文管理（AsyncLocalStorage）
- 每个请求独立配置副本
