# Python 控制器架构分析文档

## 1. 项目概述

### 1.1 定位
**AI Controller** 是 AIClient-2-API 的核心大脑，负责本地大语言模型的智能调度、GPU 资源监控和队列管理。采用 FastAPI 构建，实现"外壳解耦，内核驱动"的架构理念。

### 1.2 核心价值
- 🎯 **可靠性保障**：防止 Node.js 侧长时间挂起超时
- 🎯 **流量削峰**：利用并发控制保护 GPU，防止显存溢出
- 🎯 **自动管理**：根据请求自动启动/停止模型，节省资源
- 🎯 **智能调度**：模型热切换、显存碎片回收、Redis 队列

## 3. 核心模块分析

### 3.1 调度器 (Scheduler)

#### 职责
- 模型热切换
- 预加载策略
- 空闲超时管理
- 显存资源检查

#### 关键方法

| 方法 | 功能 |
|------|------|
| `acquire_request(model_name)` | 获取请求槽位，启动模型 |
| `release_request(model_name)` | 释放请求，检查空闲超时 |
| `start_model(model_name)` | 启动指定模型 |
| `stop_model(model_name)` | 停止指定模型 |
| `switch_model(from_model, to_model)` | 热切换模型 |
| `_check_idle_timeout()` | 检查空闲超时 |

#### 配置示例

```yaml
models:
  gemma-2-9b:
    service: vllm-gemma
    port: 8000
    required_memory: 12GB
    preload: true      # 启动时预加载
    keep_alive: true   # 持续运行不停止
```

### 3.2 GPU 监控器 (GPUMonitor)

#### 职责
- 解析 nvidia-smi 输出
- 实时采集 GPU 状态
- 显存使用监控
- 温度和利用率监控

#### 数据结构

```python
{
    "status": "available",
    "gpu_count": 1,
    "primary": {
        "name": "NVIDIA RTX 4090",
        "total_memory": "24GB",
        "used_memory": "12GB",
        "available_memory": "12GB",
        "temperature": 65,
        "utilization": 80
    }
}
```

### 3.3 限流器 (RateLimiter)

#### 职责
- Redis 队列管理
- 并发控制
- 请求状态追踪
- 队列长度监控

#### 核心数据结构

| Key | 类型 | 说明 |
|-----|------|------|
| `active_requests:{model}` | Counter | 当前活跃请求数 |
| `queue:{model}` | List | 等待队列 |
| `request:{request_id}` | Hash/String | 请求详情 |

#### 关键方法

| 方法 | 功能 |
|------|------|
| `enqueue_request(model_name, request_info)` | 入队，返回 request_id |
| `dequeue_request(model_name)` | 出队，获取请求详情 |
| `cancel_request(request_id)` | 取消排队中的请求 |
| `get_request_status(request_id)` | 查询请求状态 |
| `is_available(model_name)` | 检查是否有可用槽位 |

### 3.4 系统控制器 (SystemController)

#### 职责
- systemd 服务管理
- 服务启停控制
- 进程状态监控

### 3.5 vLLM 代理 (VLLMProxy)

#### 职责
- OpenAI 协议转换
- 流式响应代理
- 显存优化控制
- 缓存清理

#### 显存策略

| 策略 | 显存利用率 | 说明 |
|------|-----------|------|
| `conservative` | 0.8 | 保守模式，稳定优先 |
| `balanced` | 0.9 | 平衡模式（默认） |
| `aggressive` | 0.95 | 激进模式，性能优先 |

### 3.6 WebSocket 管理器 (WebSocketManager)

#### 职责
- WebSocket 连接管理
- 实时状态广播
- 连接生命周期管理

#### 推送数据结构

```python
{
    "type": "status_update",
    "timestamp": "2024-01-01T12:00:00",
    "gpu": {...},
    "models": {...},
    "queue": {...}
}
```

### 4.1 请求处理流程

```
┌─────────────┐
│  客户端请求  │
└──────┬──────┘
       ▼
┌─────────────────────┐
│  RateLimitMiddleware │ 限流检查
└──────┬──────────────┘
       ▼
┌─────────────────────┐
│  TimeoutMiddleware │ 超时控制
└──────┬──────────────┘
       ▼
┌─────────────────────┐
│   API Handler       │ 路由分发
└──────┬──────────────┘
       ▼
┌─────────────────────┐
│Scheduler.acquire()  │  获取槽位
└──────┬──────────────┘
       │
       ├───────────────┐
       ▼               ▼
  ┌─────────────┐  ┌─────────────┐
  │模型已运行? │  │模型未运行   │
  └──────┬──────┘  └──────┬──────┘
         │                │
         ▼                ▼
   ┌───────────┐   ┌───────────────┐
   │直接转发请求│   │ start_model()│
   └──────┬────┘   └───────┬───────┘
          │                │
          └───────┬────────┘
                  ▼
   ┌───────────────────────────┐
   │ VLLMProxy.chat_completion()│ 转发请求
   └───────────┬───────────────┘
               ▼
   ┌───────────────────────────┐
   │Scheduler.release_request()│ 释放槽位

               ▼
   ┌───────────────────────────┐
   │   返回响应给客户端         │
   └───────────────────────────┘
```

## 5. 设计模式应用

### 5.1 策略模式
- **应用场景**：显存优化策略（conservative/balanced/aggressive）
- **优势**：算法族可自由切换，符合开闭原则

### 5.2 单例模式
- **应用场景**：Scheduler、GPUMonitor、RedisClient 等核心组件
- **优势**：资源复用，全局状态统一管理

### 5.3 观察者模式
- **应用场景**：WebSocket 状态广播、配置热更新通知
- **优势**：松耦合，事件驱动

### 5.4 责任链模式
- **应用场景**：中间件链（RateLimit → Timeout → ErrorHandler）
- **优势**：灵活组合，易于扩展

## 7. 配置说明

### 7.1 模型配置

```yaml
models:
  {model_name}:
    service: string        # systemd 服务名
    port: int              # 监听端口
    required_memory: str   # 所需显存 (如: 12GB)
    preload: bool          # 是否预加载
    keep_alive: bool       # 是否常驻
```

### 7.2 全局设置

```yaml
settings:
  concurrency_limit: int        # 并发限制
  min_available_memory: str    # 最小可用显存
  request_timeout: int          # 请求超时(秒)
  model_start_timeout: int      # 模型启动超时(秒)
  preload_timeout: int          # 预加载超时(秒)
  idle_timeout: int             # 空闲超时(秒)
  memory_flush_interval: int    # 显存清理间隔(秒)
  default_memory_strategy: str  # 默认显存策略
```

### 7.3 环境变量

| 变量名 | 默认值 | 说明 |
|--------|--------|------|
| `REDIS_URL` | `redis://localhost:6379` | Redis 连接地址 |
| `PORT` | `5000` | 服务端口 |
| `LOG_LEVEL` | `INFO` | 日志级别 |

## 9. 性能优化

### 9.1 已实现的优化

| 优化项 | 方案 | 收益 |
|--------|------|------|
| **连接池** | HTTP 连接复用 | 减少连接开销 |
| **异步处理** | 全链路异步 | 提高吞吐量 |
| **H2 缓存** | HTTP/2 连接复用 | 降低握手开销 |
| **预加载** | 启动时加载常用模型 | 秒级响应 |

### 9.2 建议优化

| 优化点 | 当前状态 | 建议方案 | 预期收益 |
|--------|----------|----------|----------|
| **缓存策略** | 无请求缓存 | 添加响应缓存 | 降低重复计算 |
| **智能调度** | 固定并发限制 | 基于显存动态调整 | 提高资源利用率 |
| **优雅降级** | 直接拒绝请求 | 队列缓冲 + 超时 | 提升用户体验 |
